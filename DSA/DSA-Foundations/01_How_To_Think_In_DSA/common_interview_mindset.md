# Common Interview Mindset

---

## What interviewers actually evaluate

| Signal | What it means |
|--------|---------------|
| Communication | Can you explain thinking? |
| Problem decomposition | Break into steps? |
| Correctness | Edge cases handled? |
| Complexity awareness | Know trade-offs? |
| Code quality | Readable, bug-free? |

They are **not** only checking if you memorized LeetCode.

---

## WHY mindset matters

Two candidates with same optimal solution: the one who explains **why** it works gets higher hire signal.

---

## HOW to behave in interview

1. **Clarify** — "Can duplicates exist? Can I modify the array?"
2. **Think aloud** — silence is risky; narrate at a calm pace.
3. **Start simple** — brute force is OK to say first.
4. **Collaborate** — treat interviewer as teammate.
5. **Recover** — if stuck, go back to examples.

---

## WHEN to ask for hints

After 3-5 minutes stuck on one approach, say:
"I have tried X and Y. Could you confirm if Z direction is worth exploring?"

---

## Real-world analogy

Interview = driving with instructor. They want to see you check mirrors (edge cases), signal turns (explain steps), and park safely (working code).

---

## Common Mistakes

- Apologizing excessively
- Never testing code
- Arguing with hints
- Giving up without brute force attempt

---

## 1-Minute Revision

Clarify → Examples → Brute → Optimize → Code → Test → Complexity
